# 博客系统

这已经不是一个“只能发文章”的静态博客外壳，而是一套围绕内容发布、后台管理、研究工作流、评论互动、AI 工具和运维能力搭起来的完整系统。

如果只从首页看，它像一个个人博客；如果从代码和后台看，它更像一个面向单人维护者的轻量内容平台。

这篇文档把当前版本真正已经具备的能力，按“系统定位、架构、内容模型、前后台功能、研究工作流、AI 能力、运维方式”完整梳理一遍，方便后续继续维护、扩展和交接。

## 一、系统定位

这个项目目前可以同时被理解成四层能力叠加在同一个仓库中的产品：

1. 一个公开可访问的中文风格博客站点。
2. 一个以管理员为中心的内容管理后台。
3. 一个站长私有的研究工作台。
4. 一套可继续扩展的 AI 与 RAG 能力层。

也就是说，它不是“博客 + 几个附加页面”，而是把写作、发布、审核、研究、调试和运维整合到了同一套系统里。

## 二、技术栈与实现方式

当前项目的核心技术栈如下：

- 前端框架：Next.js 16 App Router
- 交互层：React 19 + TypeScript
- 样式系统：Tailwind CSS v4 + 全局主题变量
- 数据层：Prisma + PostgreSQL
- 认证方式：自定义会话体系 + HttpOnly Cookie
- 表单校验：Zod
- 邮件能力：Nodemailer
- 部署方式：Docker / Docker Compose
- 自动化：GitHub Actions + 定时脚本

![博客系统技术架构图](/docs/blog-system-architecture.svg)

上图把当前系统最核心的几层放到了一起：左侧是公开访问入口，中间是页面与业务逻辑层，右侧是后台工作台，下方则是数据库、内容模型、研究工作流和外部服务的连接关系。

这套选型的好处是，页面、API、Server Actions、数据库模型和后台工具全部集中在同一项目里，维护成本相对可控，尤其适合个人站点持续迭代。

## 三、整体架构分层

从职责上看，系统大致可以分成四层。

### 1. 展示层

主要由 `app/` 和 `components/` 构成，负责所有公开页面、后台页面以及前端交互。

这一层包含：

- 首页、博客、笔记、周报、论文、工具页等公开页面
- 管理后台的表单、列表、详情页
- 评论区、聊天浮层、主题切换、轮播图等交互组件

### 2. 应用层

主要在 `lib/` 和 `lib/actions/` 中，负责业务规则、权限判断、查询封装和写入逻辑。

这一层承担的工作包括：

- 内容查询与聚合
- 评论审核逻辑
- 管理员权限控制
- 论文同步、周报生成、RAG 同步
- 内容发布、分类管理、系列管理、调试请求等后台动作

### 3. 数据层

由 `prisma/schema.prisma` 和 PostgreSQL 组成，是系统的结构化数据中心。

数据库中已经包含比较完整的模型：

- 用户、会话、2FA、邮箱验证、密码重置
- 文章、笔记、日志、图集、系列、评论、修订记录
- 论文主题、每日论文、论文库、论文批注、周报
- RAG 知识块、查询日志
- 访问统计、搜索记录、通知、审计日志

### 4. 集成与任务层

这部分负责把系统和外部能力连接起来。

当前已经接入或预留了这些能力：

- 邮件发送
- 多模型 Provider 配置
- 转写 Provider 切换
- arXiv 论文同步
- Markdown 内容同步
- RAG 知识同步
- 周报生成脚本

## 四、核心内容模型

系统中的内容并不是只有“文章”一种。

### 1. Post：长文文章

这是博客的主体内容，通常用于：

- 长篇写作
- 系统说明
- 项目复盘
- 方法论总结
- 架构与功能文档

文章支持：

- 分类
- 标签
- 置顶
- 精选
- 系列归属
- 修订历史
- 评论互动

### 2. Note：常青笔记

适合保存可复用、可沉淀的知识片段，比如：

- 概念卡片
- 清单
- 方法模板
- 阅读摘记
- 小型知识条目

### 3. Journal：研究日志

这是更轻量、更偏时间流的更新形式，用来记录近期在做什么、读了什么、推进了什么。

### 4. Weekly Digest：周报

周报把论文同步结果、日志更新和主题内容重新组织成一份阶段性回顾，更适合做对外展示和阶段复盘。

### 5. Gallery：图集

用于承载视觉内容，不和长文写作混在一起。

### 6. Papers：论文系统

包括两部分：

- 公开页面中的每日论文雷达
- 仅管理员可见的私有论文库

前者偏展示，后者偏工作流。

## 五、前台公开功能

### 1. 首页

首页不是传统的“自我介绍 + 最新文章”模板，而是一个综合入口页。

目前首页聚合了这些信息：

- 焦点轮播
- 置顶文章
- 最新站点内容流
- 作者侧栏
- 归档入口
- 热门标签
- 最新评论

它的作用不是只放装饰信息，而是把全站不同内容形态串成一个连续的阅读入口。

### 2. 博客文章页

博客文章页支持：

- 分类浏览
- 标签跳转
- 置顶排序
- 阅读时长
- 评论区
- 相关文章与信息串联

### 3. 常青笔记 / 研究日志 / 周报 / 图集

这些模块各自独立，但又通过首页、归档、标签、分类、系列等入口重新连接起来，避免内容被切成彼此孤立的块。

### 4. 归档、分类、标签与搜索

为了让内容不只按时间排序，这个站点提供了多种发现方式：

- 归档：按时间线回看内容
- 分类：按主题组织内容
- 标签：按关键词交叉检索
- 系列：按专题顺序组织多篇内容
- 搜索：快速定位站内信息

### 5. 论文公开页

每日论文雷达已经不是简单的纯列表页，而是一个面向阅读与筛选的公开入口，适合持续追踪主题变化。

### 6. Tools 页面

当前工具页已经公开，不再局限管理员使用，适合放外链导航、工具入口和一些轻量调试能力。

## 六、后台管理能力

后台并不是只有“发文章”。

当前后台已经覆盖以下模块：

- 文章管理
- 笔记管理
- 日志管理
- 图集管理
- 分类管理
- 系列管理
- 评论审核
- 评论规则管理
- 论文主题与论文库管理
- 周报管理
- 站点资料管理
- Provider 管理
- RAG 管理
- 数据分析
- 数据导出
- 审计日志
- API 请求调试

也就是说，后台同时承担了 CMS、审核台、研究台和运维台的角色。

## 七、研究工作流能力

这一部分是这个博客和普通内容站最不一样的地方。

### 1. 每日论文雷达

系统可以围绕配置好的论文主题去同步 arXiv 内容，并在前台形成公开的论文更新页。

### 2. 私有论文库

管理员可以把论文沉淀进自己的私有库中，并继续做后续工作：

- 标记阅读状态
- 记录阅读进度
- 添加批注
- 保存引用

### 3. 周报生成

周报把论文、日志和阶段性内容重新汇总，形成可公开展示的阶段回顾。

### 4. 内容沉淀链路

当前站点已经形成一条比较清晰的内容演化路径：

1. 从论文同步开始发现材料。
2. 在论文库中进行筛选、阅读和批注。
3. 把稳定结论沉淀为 Note。
4. 把完整观点扩展为 Blog Post。
5. 用 Weekly Digest 做阶段整理。

这意味着它不仅能“发结果”，还能承接“研究过程”。

## 八、AI 与工具能力

### 1. Provider 管理

系统支持多 Provider 配置，便于切换不同模型和不同转写服务。

### 2. 聊天能力

站内有悬浮聊天入口，可以结合页面上下文进行问答，而不是完全脱离页面内容的外部聊天框。

### 3. RAG 能力

RAG 相关能力已经不是概念状态，而是具备基础操作闭环：

- 知识同步
- Chunk 管理
- 检索日志
- 来源追踪
- 公开 / 私有知识范围区分

### 4. API 调试与请求验证

后台新增的请求调试页，可以针对任意 API 进行请求验证，适合做：

- Header 调试
- Body 调试
- Provider 接口排查
- 请求结果查看

这类工具对于一个长期维护的博客系统很重要，因为它让系统不只是“能展示”，还具备自我排障能力。

## 九、写作与更新流程

这个项目支持两种内容维护方式。

### 1. 后台编辑

通过 `/admin/*` 页面直接编辑、保存和发布内容。

适合：

- 快速在线修改
- 后台集中管理
- 评论和内容一起处理

### 2. Markdown 同步

通过 `content/` 目录维护内容，再同步进数据库。

当前目录包括：

- `content/blog`
- `content/notes`
- `content/journal`

同步命令包括：

```bash
npm run content:sync
npm run content:watch
```

这种方式特别适合长期写作和版本管理。

当前版本中，博客文章的 Markdown frontmatter 也支持：

- `featured: true`
- `pinned: true`

其中 `pinned: true` 会把这篇已发布文章提升为当前置顶文章，并自动取消其他已发布置顶文章的置顶状态。

## 十、安全与访问模型

这套系统的访问模型已经比较明确：

- 前台浏览默认公开
- 管理后台仅管理员可登录
- 公共注册关闭
- 评论对游客开放
- 游客昵称必填，邮箱可选且不公开展示

安全机制包括：

- 自定义会话体系
- HttpOnly Cookie
- 邮箱验证
- 找回密码 / 重置密码
- 管理员 2FA
- 登录限流
- 异常登录提醒
- 评论反垃圾与规则审核
- 审计日志

这保证了它既保留博客的开放性，又不会把后台完全暴露成高风险入口。

## 十一、部署与运维

本地和生产的常用命令已经比较完整：

```bash
npm run dev
npm run build
npm run db:push
npm run content:sync
npm run papers:sync
npm run digest:generate
npm run rag:sync
```

生产上既可以用 Docker / Docker Compose，也可以结合 GitHub Actions 做自动化流程。

从运维角度看，这个项目已经覆盖了几个关键能力：

- 构建校验
- 数据库结构同步
- Markdown 内容同步
- 论文数据同步
- 周报生成
- RAG 数据同步

这让它不仅能“长期写”，也能“长期运转”。

## 十二、这个系统当前最有价值的地方

如果要用一句话概括它现在最重要的价值，那就是：

它把“写作、研究、审核、调试、运维”放到了同一套个人站点系统里。

这比单纯做一个外观好看的博客更有长期价值，因为它解决的是持续创作和持续维护的问题。

## 十三、后续还值得继续增强的方向

当前版本已经能用，但后面仍然有一些很自然的增强方向：

- 更细粒度的通知偏好
- 更完整的对象存储与媒体处理
- 更强的分析报表
- 更丰富的 RAG 数据来源
- 更成熟的多篇文章专题组织能力
- 更强的后台任务监控

这些都不是“推翻重做”，而是在现有架构上继续往前加深。

## 结语

如果你把它当成博客，它已经足够完整；如果你把它当成一套个人内容操作系统，它也已经有了相当扎实的骨架。

后续继续扩展时，最值得坚持的方向不是功能越多越好，而是继续保持这几点：

- 前台信息结构清晰
- 后台操作链路顺手
- 研究工作流可持续
- AI 工具真正服务内容，而不是喧宾夺主

这也是这篇文档被放到置顶位置的原因：它更能代表这个博客现在真正是什么，以及之后准备往哪里走。
